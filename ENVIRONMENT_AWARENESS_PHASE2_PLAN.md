# Connor 环境感知第二阶段修改方案

状态：已确认，实施中  
制定日期：2026-07-24  
基线分支：`air/environment-awareness-phase1`  
基线提交：`3e9b5901 feat: add per-turn environment awareness`

## 1. 目标

在第一阶段“每个用户回合获取一次当前位置和即时天气”的基础上，增加本地环境历史、按需查询、变化检测和回答相关性控制。

环境能力用于给当前任务补充必要事实，不用于分析用户本人。系统不得利用位置或环境记录推断用户的家庭、公司、常用地点、兴趣地点、生活规律、身份、健康状况或其他画像。

本文件是开发与验收规格，不注册为 LLM、Memory OS 或知识检索的数据源。

## 2. 已确认的产品规则

1. 保留第一阶段的每回合一次环境 Bootstrap；同一用户回合的内部工具循环不重复定位。
2. 继续使用固定、结构化的 Open-Meteo 接口，不使用普通网页搜索替代天气接口。
3. 不新增日历摘要，也不修改现有 Calendar Bootstrap。日历仍由现有标准启动流程负责。
4. 环境信息按用户任务指定的类别提供；普通编程、写作、总结等无关任务不主动展示环境信息。
5. 用户没有出行、户外、穿衣或健康相关任务时，不提示带伞、防晒、增减衣物等建议。
6. 出现明显或安全相关的环境变化时，可以附加简短事实提示；如果任务与行动建议无关，只陈述事实，不扩展建议。
7. 不进行地理信息学习，不运行地点聚类，不调用 LLM 推断家、公司、常用地点或兴趣地点。
8. 环境数据永久保留，不按时间自动删除，也不向用户提供编辑或删除能力。
9. 每日后台任务只做确定性的数据库维护，不调用 LLM。

## 3. 位置隐私边界

### 3.1 精确位置

- Core Location 返回的精确经纬度只在内存中短暂存在。
- 精确坐标不得写入 SQLite、文件、日志、审计正文、错误信息或遥测。
- 精确坐标不得进入 LLM 上下文、Memory OS、Session Capsule 或历史工具结果。
- 获取环境数据前，先在本地映射为约 3–5 公里的粗粒度网格，再向第三方 provider 发起请求。
- 请求结束后不保留原始精确坐标的引用或可恢复表示。

### 3.2 粗粒度位置

- 持久层只保存网格 ID、网格中心点、网格精度版本、时区以及必要的行政区域显示信息。
- 网格中心点代表区域，不代表用户的真实位置；API 和界面必须使用 `region` 或 `grid` 语义，不能命名为 `userLocation`。
- 粗粒度位置只用于关联和查询环境记录，不作为用户画像、行为分析、地点语义学习或因果推断的依据。
- 历史地点未知时，不得使用当前位置补位，也不得猜测用户当时所在区域。
- 道路级状况、附近具体实物等能力若不能由粗粒度位置可靠支持，应返回“数据不足”，不得虚构。

## 4. 回答相关性策略

运行时先判断环境信息是否与当前请求有关，再决定是否获取、展示或转化为建议。

建议使用以下类别枚举：

- `travel`
- `outdoor`
- `clothing`
- `health`
- `weather`
- `air_quality`
- `traffic`
- `local_context`
- `historical_analysis`

行为规则：

| 情况 | 行为 |
|---|---|
| 用户明确指定环境类别 | 仅提供指定类别及完成任务所需的直接关联事实 |
| 用户提出出行、户外、穿衣或健康任务 | 可以把相关环境事实转化为简短、可执行的建议 |
| 环境与任务无关 | 不在回答中展示环境信息 |
| 检测到明显或安全相关变化 | 可附加简短事实；仅在任务相关时提供行动建议 |
| 数据缺失、过期或质量不足 | 明确说明未知，不进行推断 |

变化检测必须由确定性阈值和来源数据完成，不调用 LLM。阈值应集中配置并有测试，首版至少覆盖极端温度、强降水、强风、空气质量显著恶化和 provider 发布的有效预警。

## 5. 独立环境数据库

路径：

```text
Application Support/Connor/environment/environment.sqlite
```

环境数据库与 Memory OS 图数据库分离，避免环境记录被默认当作用户知识或画像证据。

### 5.1 表组

#### `environment_schema_migrations`

记录环境数据库 schema 版本和迁移时间。

#### `environment_regions`

保存粗粒度网格，不保存原始精确位置。建议字段：

- `region_id`
- `grid_system`
- `grid_precision_version`
- `grid_cell_id`
- `center_latitude`
- `center_longitude`
- `timezone_identifier`
- `country_code`
- `administrative_area`
- `locality`
- `created_at`
- `updated_at`

唯一键：`(grid_system, grid_precision_version, grid_cell_id)`。

#### `environment_time_series_points`

保存天气时序点。建议字段包括区域、时间、数据质量、温度、体感温度、湿度、降水、天气代码、风速、阵风、云量、能见度、气压及 provider 元数据。

唯一键应至少包含：`(region_id, observed_at, data_kind, provider)`。

`data_kind` 固定为：

- `current_observation`
- `forecast`
- `historical_observation`
- `historical_reanalysis`

#### `environment_air_quality_points`

保存空气质量时序数据，与天气表采用相同的区域、时间、质量和 provider 语义。

#### `environment_alerts`

保存 provider 提供的预警、有效时间、严重级别、影响区域和状态。不得通过 LLM 自行生成官方预警。

#### `environment_notice_state`

保存变化提示的去重状态，防止同一变化在短时间内重复提示。只存环境事件和提示时间，不存用户行为结论。

#### `environment_history_queries`

保存历史区间补齐任务的技术状态，例如请求区间、provider、完成状态和错误分类。不得保存用户问题原文或 LLM 推理内容。

#### `environment_provider_state`

保存 provider 的缓存、退避、速率限制和最近成功时间等运行状态。

### 5.2 保留和维护

- 所有已成功写入的环境数据永久保留。
- 使用事务和幂等 UPSERT 控制重复数据。
- 每日维护只执行 `PRAGMA optimize`、必要的 checkpoint、重复完整性检查和失败任务恢复。
- 不执行基于记录年龄的自动删除。
- 不实现面向用户或 LLM 的编辑、按时间删除、按类别删除或全部删除接口。
- schema migration 和确定性修复程序可以纠正技术性损坏或 provider 重复记录，但不能被用作普通数据编辑入口。

## 6. 历史环境工具组

### `environment_history_coverage`

检查指定粗粒度区域、类别和时间区间的数据覆盖度、质量类型及缺口，不触发网络补齐。

### `environment_history_query`

查询指定区域和历史区间的环境事实。优先读取本地 SQLite；存在缺口且策略允许外部网络时，从固定 provider 补齐并永久缓存。

### `environment_history_compare`

对两个或多个明确区域或时间区间做确定性统计比较，返回事实、样本覆盖率和质量差异，不输出因果结论。

三个工具都必须：

- 使用 closed JSON Schema，拒绝未知字段。
- 要求调用方明确提供历史区域；不得默认使用当前位置。
- 返回数据质量：观测、预测、归档观测或历史再分析。
- 返回覆盖率、provider 和时间边界。
- 不返回精确位置，不接受精确位置参数。
- 不根据环境记录推断用户偏好、健康、行为或历史地点。

## 7. Prompt 与 Memory OS L1 集成

### 7.1 主会话 System Prompt

只加入跨工具策略，不复制参数手册：

- 仅当用户请求与当前或历史环境相关时调用环境工具。
- 无关任务不展示环境信息。
- 历史地点必须来自用户明确指定或任务中已有的可靠地点证据。
- 缺少历史地点时先询问或说明无法查询，禁止以当前位置代替。
- 环境相关性不等于因果关系。
- 行动建议必须同时满足任务相关性与数据质量要求。

### 7.2 Memory OS L1 后台工具

为 L1 知识提取过程注册历史环境只读工具，但设置严格边界：

- 仅在分析的历史任务明确包含环境条件时调用。
- 环境记录只能作为该任务当时的背景证据，不能成为用户画像事实。
- 不把相关性提升为因果事实。
- 不推断用户的历史地点、偏好、健康或生活规律。
- 不将粗粒度网格写入人物、组织、地点兴趣或关系知识。
- 工具返回的环境事实必须保留来源、时间、区域精度和质量标记。

## 8. 后台并发与性能

- Core Location 对象生命周期继续遵守 Apple 框架的 actor 要求。
- 定位等待、provider 网络请求、JSON 解码、数据库读写、历史补齐和每日维护均不得阻塞主线程。
- UI 只接收不可变结果或状态快照。
- 同一区域、类别和时间区间的并发补齐请求应合并，避免重复网络调用。
- 保留第一阶段的超时、15 分钟缓存和旧缓存降级；数据库缓存与内存缓存职责分开。

## 9. Provider 策略

- 即时天气继续使用 Open-Meteo Forecast API。
- 历史天气使用 Open-Meteo Historical Weather API；其结果按 provider 定义标记为归档观测或历史再分析，不伪装成现场观测。
- 空气质量使用 Open-Meteo Air Quality API，并保留其数据来源和时间粒度。
- 路况、风俗、周围实物在没有稳定、结构化且许可明确的数据源前，不纳入第二阶段核心实现。
- provider 请求只使用粗粒度网格中心点。
- provider 失败时优先返回带时间戳的本地数据，并明确标记陈旧状态；不自动回退到不受控网页内容。

## 10. 实施顺序

1. 在当前 `air/environment-awareness-phase1` 分支继续实施。
2. 定义环境领域模型、网格化函数、数据质量枚举和 repository 协议。
3. 创建独立 SQLite 数据库、迁移系统、表组和索引。
4. 调整第一阶段 provider 边界：精确坐标在本地网格化后再发起网络请求，并确保日志无坐标泄漏。
5. 实现即时数据持久化、幂等 UPSERT 和 provider 状态管理。
6. 实现历史天气与空气质量 provider、覆盖检查和缺口补齐。
7. 实现三个历史环境工具及严格 schema。
8. 实现确定性的任务相关性门控、变化检测和提示去重。
9. 更新主会话 System Prompt，并向 L1 后台流程注册受限只读工具。
10. 实现每日数据库维护任务。
11. 更新工程说明和隐私说明。
12. 每完成一个实施阶段分别提交；完成后执行一次集中验证，不自动推送远端。

## 11. 验收标准

- 任意持久化文件、日志和工具结果中不存在 Core Location 原始精确坐标。
- 发给环境 provider 的坐标已经映射为约 3–5 公里网格中心点。
- 普通编程、写作和总结任务不会出现无关天气或生活建议。
- 无出行等相关任务时，即使发生变化也只提供必要事实，不自动提示带伞或防晒。
- 历史工具能报告覆盖、查询和比较，并正确区分四类数据质量。
- 历史地点缺失时调用失败或要求补充，不会使用当前位置替代。
- L1 可以在明确相关时读取环境背景，但不会产生地点学习、用户画像或因果断言。
- 数据库记录可幂等写入且永久保留，不存在面向用户或 LLM 的编辑、删除工具。
- 网络、数据库和历史补齐工作不阻塞主线程。
- 现有日历 Bootstrap 行为和第一阶段每回合一次定位语义保持不变。

## 12. 明确不做

- 保存精确位置历史。
- 向用户或 LLM 提供环境记录编辑、删除或清除能力。
- 推断或标注家、公司、常用地点和兴趣地点。
- 每日调用 LLM 分析位置或环境记录。
- 根据环境记录建立用户画像或行为模型。
- 把当前位置当作任意历史任务的位置。
- 在没有可靠数据源时声称知道道路级路况、附近具体实物或当地风俗。
- 修改现有日历摘要和 Calendar Bootstrap。
- 将本方案文档本身作为 LLM 或 Memory OS 的分析依据。
